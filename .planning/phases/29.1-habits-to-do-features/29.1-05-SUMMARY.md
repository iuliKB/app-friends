## Decisions Made

- **Detail screen calls `toggle_habit_today_checkin` directly** instead of using `useHabits().toggleToday`. The plan §<action> said "Prefer the latter [adding toggle to useHabitDetail] for cleaner separation" but mounting `useHabits()` on the detail screen as a secondary option would have triggered an additional `get_habits_overview` round-trip per detail render — wasteful when `useHabitDetail.refetch()` already reloads exactly the data the detail screen displays. Direct RPC keeps the detail screen's network surface minimal.
- **Invitations rendered as `ListHeaderComponent`** rather than promoting to a `SectionList`. The two row shapes (HabitInvitationRow card vs HabitRow row) and the two empty states (zero invites is invisible; zero habits shows the main EmptyState) don't fit SectionList's symmetric model. ListHeaderComponent lets us drop the entire invites block when empty without inventing a synthetic "main habits" section header that has no UI-SPEC copy.
- **Friend list lazy-loaded in create.tsx** — only fetches when user taps "With friends". Reason: per CONTEXT.md, Solo is the expected default ("user's first habit"); pre-fetching friends on every visit to the create screen wastes a round-trip when the majority of submits will be solo. Trade-off: a tiny delay (one ActivityIndicator render) when the user first switches to group mode. Marginal and matches the project's other lazy-loaded composer flows.
- **`(supabase as any)` cast at the 3 new RPC sites and 2 new table query sites**, matching the existing pattern in `useChatRoom.ts:612` and `useHabitDetail.ts:60-72`. The Plan 03 SUMMARY explicitly anticipates this (line 161-164): `database.ts` predates migration 0024, and regenerating it is out-of-scope for Wave 3 — Plan 06 will need the cast pattern too, and a batched regen at the end of the phase will be cleaner than touching `database.ts` in two separate plans.
- **Skeleton row height set to 56 in index.tsx** to match `HabitRow.minHeight: 56` (UI-SPEC §Layout — row min-height touch-friendly). A different skeleton height would cause a visible jitter when real rows render and replace skeletons.
