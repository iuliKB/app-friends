<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| User input → create_habit RPC | Title, cadence, weekly_target, member_ids are user-controlled |
| Friend picker → member_ids | UI selects from useFriends() output — could be tampered via direct rpc call (bypassing UI) |
| Pending invitation accept → habit_members.accepted_at update | User updates their own row; RLS gates user_id = auth.uid() |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-29.1-23 | Tampering | Attacker submits create_habit with p_member_ids containing non-friend UUIDs | accept | Server has no enforcement that members must be friends (group habits work with anyone the user knows the UUID of). Friend list is a UX filter only. Future: add friendship check inside create_habit RPC if abuse emerges. |
| T-29.1-24 | Tampering | Attacker accepts an invitation they weren't sent | mitigate | `habit_members` UPDATE policy requires `user_id = auth.uid()` (Plan 01). Acceptance criteria verified in Plan 01. |
| T-29.1-25 | Info Disclosure | Habit detail screen leaks members' avatars to non-members | mitigate | `useHabitDetail` queries through RLS — non-accepted members cannot SELECT habit_checkins. Pattern: caller must be accepted member of habit to see members at all. |
| T-29.1-26 | DoS | Attacker creates 10,000 habits flooding storage | accept | Free-tier hard caps; no abuse vector in friend-group app. Soft cap: UI could limit to ~50 habits per user but not enforced in v1. |
| T-29.1-27 | Spoofing | Habit creator field falsified | mitigate | `create_habit` RPC sets `created_by := auth.uid()` from session; cannot be forged by client. |
</threat_model>
