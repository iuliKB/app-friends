# Phase 29.1: Habits & To-Do Features - Research

**Researched:** 2026-05-12
**Domain:** Social feature schema + atomic RPCs + Bento Hub extension + chat attachment + home widgets
**Confidence:** HIGH (every recommendation is grounded in an in-repo precedent that already shipped)

## Summary

Phase 29.1 adds two ledger-style social features (Habits and To-Dos) and a chat to-do bubble. Every architectural decision in this phase has a clean precedent already in the repo — there is **no greenfield exploration to do**. The work is almost entirely *clone-and-adapt* from the IOU phase (8/9), the Birthday Social phase (11), the Polls phase (17), and the Bento Hub commit (`5caf16a`).

The two highest-risk findings — and the things the planner must surface to the user before locking the plan — are:

1. **System messages do NOT exist yet.** CONTEXT.md asserts that "plan created system messages" exist as the model for D-11's completion roundtrip. They do not. `MessageType = 'text' | 'image' | 'poll' | 'deleted'` (verified in `src/types/chat.ts:1` and migration `0018`). Phase 29.1 must (a) extend the `message_type` CHECK constraint to add `'system'`, (b) update the `MessageType` TS union, (c) add a `MessageBubble` rendering branch for `system`, and (d) ensure RLS allows a SECURITY DEFINER RPC to insert system rows on a chat the assignee is a member of — even when the assignee is the message sender (or use a service role / bypass via the RPC).
2. **There are TWO independent Bento implementations.** The "Activity tab Bento" (`src/components/squad/bento/`) is the one Phase 29.1 must extend. There is a *second*, look-alike "tile" cluster on the Home screen inside `YourZoneSection.tsx` — these are NOT shared components. D-19 says "home gets both a Habits and a To-Dos widget"; the planner must choose one of: (a) drop new widgets into `YourZoneSection.tsx` as additional sub-tiles in the same neon-green visual language, or (b) introduce a new widget row component above/below `YourZoneSection`. Either is fine, but they are not interchangeable with `BentoCard` and copying the Activity Bento aesthetic into Home will create visual drift from Phase 29's just-shipped neon-green design.

**Primary recommendation:** Use the IOU/Birthday-Social migration template verbatim for schema. Add a single migration `0024_habits_todos_v1_8.sql` that creates all 6 tables + 5 RPCs in one file. Mirror `BentoCard`/`tileAccents` for the two new Activity tiles. Wire `'system'` message type into `MessageBubble` with a dedicated centered render branch. Two-step chat picker uses the same `Modal + Animated.timing(translateY)` pattern as `PollCreationSheet.tsx`.

### Pattern 1: Atomic SECURITY DEFINER RPC for create-with-fan-out
**What:** Single plpgsql function that does membership check → INSERT parent row → loop INSERT children.
**When to use:** `create_habit` (with optional invitees), `create_chat_todo_list` (atomic creation of list + N items).
**Source:** `supabase/migrations/0017_birthday_social_v1_4.sql:371-408` (`create_birthday_group`); `supabase/migrations/0015_iou_v1_4.sql:118-170` (`create_expense`).

```sql
-- Pattern verified from create_birthday_group (migration 0017)
CREATE OR REPLACE FUNCTION public.create_habit(
  p_title         text,
  p_cadence       text,                -- 'daily' | 'weekly' | 'n_per_week'
  p_weekly_target smallint DEFAULT NULL,
  p_member_ids    uuid[] DEFAULT NULL  -- NULL or empty = solo habit
)
RETURNS uuid
LANGUAGE plpgsql SECURITY DEFINER
SET search_path = ''
AS $$
DECLARE
  v_caller   uuid := auth.uid();
  v_habit_id uuid;
  v_member   uuid;
BEGIN
  IF v_caller IS NULL THEN
    RAISE EXCEPTION 'not authenticated';
  END IF;
  IF p_cadence = 'n_per_week' AND (p_weekly_target IS NULL OR p_weekly_target < 1 OR p_weekly_target > 7) THEN
    RAISE EXCEPTION 'weekly_target must be 1-7 for n_per_week cadence';
  END IF;

  INSERT INTO public.habits (created_by, title, cadence, weekly_target)
  VALUES (v_caller, p_title, p_cadence, p_weekly_target)
  RETURNING id INTO v_habit_id;

  -- Always add caller as a member (solo habits = creator-only member)
  INSERT INTO public.habit_members (habit_id, user_id) VALUES (v_habit_id, v_caller);

  -- Fan-out invitees (skip caller dup)
  IF p_member_ids IS NOT NULL THEN
    FOREACH v_member IN ARRAY p_member_ids LOOP
      IF v_member <> v_caller THEN
        INSERT INTO public.habit_members (habit_id, user_id) VALUES (v_habit_id, v_member);
      END IF;
    END LOOP;
  END IF;

  RETURN v_habit_id;
END;
$$;

GRANT EXECUTE ON FUNCTION public.create_habit(text, text, smallint, uuid[]) TO authenticated;
```

### Pattern 2: SECURITY DEFINER helper to break RLS recursion
**What:** Tiny `STABLE SECURITY DEFINER` function that returns boolean, used inside RLS policies that would otherwise reference the same table they protect.
**When to use:** `is_habit_member(habit_id)` for `habit_checkins` SELECT policy; `is_chat_todo_list_owner(list_id)` for `chat_todo_items` UPDATE.
**Source:** `supabase/migrations/0015_iou_v1_4.sql:60-82` (`is_iou_member`); `supabase/migrations/0017_birthday_social_v1_4.sql:127-139` (`is_group_channel_member`).

```sql
CREATE OR REPLACE FUNCTION public.is_habit_member(p_habit_id uuid)
RETURNS boolean LANGUAGE sql STABLE SECURITY DEFINER SET search_path = ''
AS $$
  SELECT EXISTS (
    SELECT 1 FROM public.habit_members
    WHERE habit_id = p_habit_id AND user_id = (SELECT auth.uid())
  );
$$;
```

### Pattern 3: Idempotent daily check-in via `UNIQUE (habit_id, user_id, date_local)` + `ON CONFLICT DO NOTHING`
**What:** Daily uniqueness lives in the schema, not in the RPC body. Tap the toggle twice in the same day → second tap is silently a no-op (or removes the row, depending on D-07 semantics).
**When to use:** `complete_habit_today` RPC.
**Source:** new pattern — no exact precedent in repo, but the closest cousin is `wish_list_claims` composite PK (`migration 0017:78-83`).

```sql
CREATE TABLE public.habit_checkins (
  habit_id   uuid NOT NULL REFERENCES public.habits(id) ON DELETE CASCADE,
  user_id    uuid NOT NULL REFERENCES auth.users(id) ON DELETE CASCADE,
  date_local date NOT NULL,                    -- caller-provided so client tz wins
  created_at timestamptz NOT NULL DEFAULT now(),
  PRIMARY KEY (habit_id, user_id, date_local)
);

-- RPC body for toggle semantics (insert if missing, else delete)
INSERT INTO public.habit_checkins (habit_id, user_id, date_local)
VALUES (p_habit_id, v_caller, p_date_local)
ON CONFLICT (habit_id, user_id, date_local)
DO NOTHING;
```

### Pattern 4: System message inserted by RPC inside same txn as the state change
**What:** `complete_chat_todo` RPC updates `chat_todo_items.completed_at`, then inserts a row into `messages` with `message_type='system'` and `body='✓ Sam completed Buy bread'`. Both succeed or both fail.
**When to use:** D-11 completion roundtrip.
**Source:** No existing system-message precedent; pattern derived from `create_poll` RPC which atomically inserts message + poll + options (`supabase/migrations/0018_chat_v1_5.sql` plus `useChatRoom.sendPoll`).

```sql
-- Inside complete_chat_todo RPC, after UPDATE chat_todo_items:
INSERT INTO public.messages (
  group_channel_id, sender_id, body, message_type
) VALUES (
  v_group_channel_id, v_caller, '✓ ' || v_assignee_name || ' completed ' || v_title, 'system'
);
```

### Pattern 5: Two-step bottom sheet via `Modal + Animated.timing(translateY)`
**What:** Single full-screen `Modal` with two view states; sheet slides up via `Animated.Value` translateY. State 1 = picker (single vs list), State 2 = form.
**When to use:** Chat "To-Do List" attachment two-step flow (D-09).
**Source:** `src/components/chat/PollCreationSheet.tsx` (full pattern); `src/components/chat/SendBar.tsx:341-372` (attachment menu modal).

### Pattern 6: Optimistic UI with revert-on-error
**What:** Update local state immediately; on error response, restore the captured snapshot.
**When to use:** habit done-toggle, to-do completion checkbox, chat to-do bubble check.
**Source:** `src/hooks/useChatRoom.ts:439-498` (sendMessage + retryMessage); same hook's addReaction (`:667-742`) is the most defensive snapshot+revert pattern in the repo.

### Anti-Patterns to Avoid
- **Storing `weekly_target` as `numeric` or `int`** — use `smallint`. Per PROJECT.md "integer storage for monetary-like things" plus CONTEXT.md note. `smallint` saves 2 bytes/row and matches `birthday_year` precedent.
- **Computing "today" in a Postgres function** — let the client pass `p_date_local date`. Server `CURRENT_DATE` is server-tz-relative and produces off-by-one in negative-UTC zones (PROJECT.md key decision: birthdays as separate columns). The same lesson applies here.
- **Triggers that insert system messages** — use the RPC. Per v1.3 outbox decision: never call extra logic inside a trigger when the user is waiting on the response.
- **Per-habit Realtime channels** — exhausts free-tier. Filter via `user_id IN (...)` on a single per-user channel.
- **Putting Habits/To-Dos under Profile** — D-20 explicitly rejects.
- **JSONB broadcast** for tile metrics — recompute from rows on `postgres_changes` events (v1.5 reactions decision).
- **Pretending system messages already exist** — they do not (see Open Questions). The plan must include the migration that adds `'system'` to the `message_type` CHECK constraint.

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Daily idempotent check-in | per-RPC `IF EXISTS … RETURN` boilerplate | `UNIQUE (habit_id, user_id, date_local)` + `ON CONFLICT DO NOTHING` | Schema-level uniqueness is race-free; RPC body is shorter |

## Common Pitfalls

### Pitfall 1: System message type does not exist — silently breaks the bubble render
**What goes wrong:** Implementer writes `INSERT INTO messages (..., message_type) VALUES (..., 'system')`. The CHECK constraint at `migration 0018` line 24 (`message_type IN ('text', 'image', 'poll')`) rejects it with a generic constraint-violation error.
**Why it happens:** CONTEXT.md asserts "plan created system messages" exist. They do not — verified by `src/types/chat.ts:1` (`MessageType = 'text' | 'image' | 'poll' | 'deleted'`) and `migration 0018:23-24`.
**How to avoid:** Migration must include:
```sql
ALTER TABLE public.messages DROP CONSTRAINT messages_message_type_check;
ALTER TABLE public.messages ADD CONSTRAINT messages_message_type_check
  CHECK (message_type IN ('text', 'image', 'poll', 'system'));
```
Plus update `MessageType` TS union, plus add a render branch in `MessageBubble.tsx` (centered, italic, no avatar, no long-press menu — long-press handler at line 458 should bail when `isSystem`).
**Warning signs:** Bubble renders with empty body; or RPC fails with `messages_message_type_check`.

### Pitfall 2: Group habit RLS recursion (`habit_members` selects from `habit_members`)
**What goes wrong:** A naïve SELECT policy `EXISTS (SELECT 1 FROM habit_members WHERE habit_id = id AND user_id = auth.uid())` triggers infinite recursion when applied to `habit_members` itself.
**Why it happens:** Same pattern as IOU members — RLS policy references the table it protects.
**How to avoid:** SECURITY DEFINER helper `is_habit_member(uuid)` (Pattern 2 above). Mirror `is_iou_member` exactly.
**Warning signs:** `infinite recursion detected in policy for relation "habit_members"`.

### Pitfall 3: Stale-closure bugs in optimistic toggle
**What goes wrong:** Done-toggle button captures stale state in its onPress closure, then setState merges with old data.
**Why it happens:** The toggle handler reads `habit.checkins` from a ref or closure that doesn't update.
**How to avoid:** Mirror `useChatRoom.addReaction` (`useChatRoom.ts:670-680`) — use `setState((prev) => ...)` to read latest state inside the updater function. Capture snapshot for rollback inside the same updater.
**Warning signs:** Tap toggle, see UI flicker between checked/unchecked because two stale updates collide.

### Pitfall 4: Realtime budget exhaustion via per-habit channels
**What goes wrong:** Client subscribes to `habit-${habitId}` for each habit on the screen → 10 habits = 10 channels → 30 friends × 10 = 300 connections → blows past free-tier limit of 200.
**Why it happens:** Implementer reaches for the obvious "channel-per-resource" mental model.
**How to avoid:** Use the existing per-user channel from `useHomeScreen.subscribeRealtime` style — single `supabase.channel('user-${userId}-checkins')` with `filter: \`user_id=in.(${memberIds.join(',')})\`` on `habit_checkins` and on `chat_todo_items`.
**Warning signs:** Realtime errors after a fan-out friend group hits ~10-12 active members; Supabase dashboard shows connection-limit warnings.

### Pitfall 5: Off-by-one date for habit check-ins in non-UTC timezones
**What goes wrong:** RPC computes `CURRENT_DATE` in server tz (UTC) → user in UTC-7 marks done at 11pm local (6am UTC next day) → row stored as tomorrow's date → tile shows "0/1 done today" all night.
**Why it happens:** Same root cause as the v1.4 birthday off-by-one (PROJECT.md decision).
**How to avoid:** Client passes `p_date_local date` parameter computed via `new Date().toISOString().slice(0, 10)` is WRONG (still UTC); use `new Date().toLocaleDateString('en-CA')` or build manually from `getFullYear/getMonth/getDate`.
**Warning signs:** Done-toggle "succeeds" but tile metric doesn't update; users in negative-UTC zones report consistently broken streak counts.

### Pitfall 6: BentoGrid layout breaks at 6 tiles + spotlight
**What goes wrong:** `BentoGrid.tsx:20` hard-codes `ROWS = 3` (spotlight + 2 grid rows). Adding 2 tiles = 6 total in 2-per-row pattern requires `ROWS = 4` (spotlight + 3 grid rows). The stagger animation array also needs to be length 4.
**Why it happens:** The grid was not designed to grow beyond 4 tiles.
**How to avoid:** Update `ROWS = 4`; add a third `Animated.View` row with `<HabitsTile />` + `<TodosTile />`; verify `staggerDelay` token from `ANIMATION.duration` still produces a coherent entrance with 4 elements.

### Pitfall 7: Bento aesthetic creeping into Home widgets
**What goes wrong:** Implementer reuses `BentoCard` directly inside `YourZoneSection.tsx` because it "looks similar" — Home now has both gradient charcoal Bento tiles and neon-green Phase-29 tiles side by side.

### Pitfall 8: Idempotent check-in semantics ambiguity (D-07)
**What goes wrong:** D-07 says "one tap toggles done" — but doesn't specify whether *un-tapping* deletes the check-in row. If the RPC only INSERTs, the user can never undo a mistaken check-in.
**Why it happens:** Spec gap.
**How to avoid:** RPC should INSERT on first call, DELETE on second call (within the same `date_local`). Use `RETURNING` to detect insert success vs no-op, or two separate RPCs (`check_in_habit_today` / `uncheck_habit_today`). Recommend: single `toggle_habit_today_checkin(p_habit_id, p_date_local)` that inspects existence and DELETEs or INSERTs accordingly, returning the resulting boolean.
**Warning signs:** Users complain they "can't un-do a tap"; QA test "tap twice in same day" fails.

### Pitfall 9: Chat to-do bubble can't long-press for retry like text bubbles
**What goes wrong:** `MessageBubble.handleLongPress` (line 458) bails on `message.pending` and on `isPoll && !isOwn`. To-do bubbles need their own bail rules — long-pressing a chat to-do should NOT show the text-message context menu (Reply/Copy/Delete).
**Why it happens:** Adding a new bubble variant requires guarding the long-press handler.
**How to avoid:** Add `if (isTodo) return;` near line 463 (mirroring poll guard). Consider whether to-dos need their own long-press menu (e.g., "Mark as done", "Reassign") — if yes, branch into a custom menu.

### Pitfall 10: System messages bypass `messages.sender_id` semantic
**What goes wrong:** System message inserted with `sender_id = caller_id` makes it look like the user said "✓ Sam completed Buy bread" — and `MessageBubble` will render it as the user's own bubble (right-aligned, orange) unless the system branch is checked first.
**Why it happens:** Existing `MessageBubble` decides own vs others purely on `sender_id === currentUserId`. The render order is poll-check → own/others.
**How to avoid:** Add `const isSystem = message.message_type === 'system'; if (isSystem) return <SystemMessageRow ... />;` BEFORE the `isPoll` check at line 587. System messages always render centered, no avatar, italic, smaller font.

```sql
CREATE OR REPLACE FUNCTION public.create_birthday_group(
  p_name       text,
  p_member_ids uuid[]
)
RETURNS uuid LANGUAGE plpgsql SECURITY DEFINER SET search_path = ''
AS $$
DECLARE
  v_caller   uuid := auth.uid();
  v_group_id uuid;
  v_member   uuid;
BEGIN
  IF v_caller IS NULL THEN RAISE EXCEPTION 'not authenticated'; END IF;
  INSERT INTO public.group_channels (name, created_by) VALUES (p_name, v_caller) RETURNING id INTO v_group_id;
  INSERT INTO public.group_channel_members (group_channel_id, user_id) VALUES (v_group_id, v_caller);
  FOREACH v_member IN ARRAY p_member_ids LOOP
    IF v_member <> v_caller THEN
      INSERT INTO public.group_channel_members (group_channel_id, user_id) VALUES (v_group_id, v_member);
    END IF;
  END LOOP;
  RETURN v_group_id;
END;
$$;
GRANT EXECUTE ON FUNCTION public.create_birthday_group(text, uuid[]) TO authenticated;
```

```typescript
async function toggleHabitToday(habitId: string, dateLocal: string) {
  let preSnapshot: HabitCheckin[] = [];
  setHabits((prev) => {
    const habit = prev.find((h) => h.id === habitId);

    return prev;

  }
}
```

## Open Questions

1. **System messages don't exist — confirm the migration approach is acceptable**
   - What we know: `MessageType = 'text' | 'image' | 'poll' | 'deleted'`; CHECK constraint at migration `0018:23-24` enforces exactly those three (`'deleted'` is set via UPDATE, not INSERT, hence not in CHECK).
   - What's unclear: Whether the user prefers `'system'` as a 5th `message_type`, OR a separate `system_messages` table, OR repurposing `'text'` with a metadata flag.
   - Recommendation: Extend `message_type` CHECK to include `'system'`. Smallest delta. Maintains single message stream for chat ordering. Surface this to user in plan-check.

2. **What happens to a chat to-do when the chat itself is deleted?**
   - What we know: `group_channels` has `ON DELETE CASCADE` on `group_channel_members` and `messages`. Chat to-do tables aren't created yet.
   - What's unclear: Should `chat_todo_lists.group_channel_id REFERENCES group_channels(id) ON DELETE CASCADE` also cascade the to-dos? Or do "From chats" items survive chat deletion as orphans pinned to the assignee?
   - Recommendation: CASCADE — chat-origin to-dos are intrinsically tied to the conversation. If user wants persistence, they can convert to "Mine" (future feature). Surface this to user in plan-check.

3. **Group habit invitation acceptance flow — inline accept or separate Pending section?**
   - What we know: CONTEXT.md `<deferred>` says "invited member sees pending group habit in their Habits list and accepts inline (no separate push)".
   - What's unclear: Schema needs a `status` column on `habit_members` (`pending`/`accepted`/`declined`)? Or use a separate `habit_invitations` table?
   - Recommendation: Single `habit_members.accepted_at timestamptz NULL` column. NULL = pending, non-NULL = accepted. Decline = DELETE row. Tile shows "X/Y done today" using only accepted members. Surface to user in plan-check.

4. **Habit "streak about to break" Spotlight signal — what defines "about to break"?**
   - What we know: D-16 says "habit about to break streak" is a Spotlight signal, but no threshold is given.
   - Recommendation: For Daily habits — missed yesterday but not today. For Weekly — current week's `weekly_target` not yet met and it's >day-5-of-7. Codify deterministic rules in `useSpotlight` extension. Surface to user in plan-check.

5. **Should the "✓ Sam completed Buy bread" system message support reactions?**
