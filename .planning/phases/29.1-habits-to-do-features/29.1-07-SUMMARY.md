## Decisions Made

- **ChatTodoBubble owns its own list-expand state internally** (not controlled-prop). Different from Plan 06's ChatTodoListRow because the chat bubble is a leaf in the FlatList — there's no parent that needs to coordinate fetches across rows. Items are passed in pre-fetched as a prop, so expand/collapse is a pure visual toggle and doesn't trigger any network or cache change. Internal `useState` keeps the bubble self-contained.
- **Lazy-fetch of chat_todo_list + chat_todo_items lives in ChatRoomScreen, not in a hook.** The plan's `<action>` explicitly admits this is the v1 placement ("a more refined solution would lift it into useChatRoom"). Keeping it on the screen avoids modifying useChatRoom (which Plan 03's SUMMARY already flagged for a latent setState-snapshot bug), and the per-message cache map + in-flight Set is a small enough surface to manage inline.
- **`useGroupMembers` extracted as a new shared hook** (not duplicated into ChatTodoPickerSheet). Pattern mirrors `GroupParticipantsSheet.fetchParticipants` but exposed reactively. Already consumed by ChatRoomScreen; future consumers (e.g. a follow-up "Reassign chat to-do" sheet) can reuse.
- **Two distinct a11y labels for assignee vs non-assignee checkbox** ("Mark done checkbox" vs "Not yet completed"). First pass used "Tap to mark done" / "Done" which collided with the bubble outer's `accessibilityLabel` containing the phrase "Tap to mark done" — `getByLabelText(/Tap to mark done/)` returned multiple matches. Renamed to give each accessible element a unique label.
- **Modal-based assignee picker uses TouchableWithoutFeedback nesting** to absorb taps inside the sheet (so taps on member rows don't bubble up to the backdrop's `onPress={() => setAssigneePickerOpen(false)}`). Same pattern as PollCreationSheet's backdrop guard.
- **`(supabase as never as <inline-shape>)` cast for chat_todo_lists / chat_todo_items reads in ChatRoomScreen** — same anticipated pattern from Plans 03/05/06 (database.ts predates migration 0024). Inline shape declared locally rather than re-typing through `any` so the result's `data` field is still typed at the call site.
- **`isSystem` and `isTodo` checks in handleLongPress occur BEFORE `isPoll && !isOwn`** in the bail chain. The order doesn't actually matter for correctness (each branch short-circuits independently), but matches the render-branch order convention so future maintainers see the same priority cascade in both code paths.
