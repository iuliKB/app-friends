## Decisions Made

- **Ref-mirrored state for optimistic snapshot capture** (vs. plan-draft setState-updater capture). The plan-draft `let snap=null; setX((prev)=>{snap=...;return prev}); if (!snap) return` pattern, cloned from `useChatRoom.addReaction:670-742`, is unreliable in React 18 because functional setState callbacks may be deferred past the next statement, leaving `snap` null when we read it. I switched both `useHabits.toggleToday` and `useTodos.completeTodo` to `habitsRef`/`mineRef` patterns that read the latest committed state synchronously. **The existing useChatRoom code likely has the same latent bug — `addReaction` returning `{ error: 'no-snapshot' }` silently — but that's outside this plan's scope.** Documented as a Rule 1 deviation.
- **Auth-store import path** is `@/stores/useAuthStore` (the actual project path), not the plan-draft `@/store/auth`. The plan's draft used a placeholder path; I matched what other hooks use (verified against `useIOUSummary`, `useHomeScreen`, `useFriends`, `useStreakData`, `useUpcomingBirthdays`). Rule 3 deviation.
- **Realtime filter is `user_id=eq.${userId}`** rather than `user_id=in.(<member_ids>)`. `HabitOverviewRow` from Plan 01's `get_habits_overview` does not include `member_ids` (it returns `members_total` count only), so the broader IN-filter isn't reachable from this hook surface. The plan-text in §<action> already acknowledged this fallback ("Realtime can simply filter on user_id IN (<caller>)"). Co-member group-habit updates surface via the next manual refetch (screen focus / pull-to-refresh) — acceptable cost vs. a per-group channel.
- **`isHabitAboutToBreak` is a pure helper inside `useSpotlight.ts`**, not an exported `useUrgentHabits` hook. The plan §<behavior> mentioned the latter, but since it's only consumed inside `selectSpotlight` and operates on the already-fetched HabitOverviewRow[], a pure function is simpler and matches the function-style of the rest of `selectSpotlight`.
- **BentoGrid + SpotlightTile follow-through deferred to Plan 04.** The plan body explicitly says (line 997): "Wave 2 Plan 04 will update BentoGrid to pass them; this plan handles the function-signature side only." Two new TS errors result in source code (`BentoGrid.tsx:24` missing args, `SpotlightTile.tsx:17` missing icon map keys); both are explicitly anticipated by the plan.

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 1 — Bug] Snapshot capture switched from setState-updater pattern to ref-mirrored state**
- **Found during:** Task 3 (test "completeTodo persists newly set completed_at when update succeeds" failed despite the UPDATE mock returning no error).
- **Issue:** The plan-draft pattern `let snap=null; setX((prev)=>{snap=prev.find(...);return prev}); if (!snap) return { error: 'X-not-found' }` relies on the setState callback running synchronously before the `if` line executes. In React 18 (and any environment that batches state updates), functional setState callbacks may be deferred — leaving `snap === null` when the `if` check fires, causing every invocation to short-circuit with `'X-not-found'`. The same pattern in `useChatRoom.addReaction:670-742` is likely silently broken in production (it just returns `{ error: ... }` and the UI never sees the optimistic update; tests pass for the revert case because revert == initial state).
- **Fix:** Added `habitsRef`/`mineRef` (`useRef<T[]>([])`) that mirrors the latest committed state. Update the ref synchronously inside every `setHabits`/`setMine` call site (refetch, optimistic flip, revert). The snapshot read becomes `xRef.current.find(...) ?? null` — synchronous, no callback timing.
- **Files modified:** `src/hooks/useHabits.ts`, `src/hooks/useTodos.ts`
- **Verification:** All 14 hook tests pass after the fix (3 useHabits + 3 useTodos + 4 useChatTodos + 4 dateLocal). The previously-failing "completeTodo persists newly set completed_at" case now passes — confirming the optimistic flip actually applies before the UPDATE resolves.
- **Committed in:** `43f0105` (useHabits fix) + `41ac776` (useTodos lands with the fix baked in).

**2. [Rule 3 — Blocker] Auth-store import path corrected from `@/store/auth` to `@/stores/useAuthStore`**
- **Found during:** Task 1 (writing the Wave 0 test scaffolds).
- **Issue:** Plan §<action> example code used `import { useAuthStore } from '@/store/auth'`, but the actual project path is `@/stores/useAuthStore` (verified against `useIOUSummary.ts:9`, `useHomeScreen.ts:3`, etc.). Using the plan-draft path would yield "Cannot find module" errors on import.
- **Fix:** All 4 new hooks + all 4 new hook test mocks use `@/stores/useAuthStore`.
- **Files modified:** `src/hooks/useHabits.ts`, `src/hooks/useHabitDetail.ts`, `src/hooks/useTodos.ts`, `src/hooks/__tests__/useHabits.test.ts`, `src/hooks/__tests__/useTodos.test.ts`, `src/hooks/__tests__/useChatTodos.test.ts`
- **Verification:** All hooks import resolves; tests run.
- **Committed in:** `ff425c6` (tests) + `4ba5a55` (hooks).

**3. [Rule 3 — Blocker] Realtime filter narrowed from `user_id IN (member_ids)` to `user_id=eq.${userId}`**
- **Found during:** Task 2 (implementing useHabits subscribeRealtime).
- **Issue:** Plan §<behavior> says "filtered by `user_id IN (<all member ids of all caller's habits>)`", but `HabitOverviewRow` (from Plan 01's `get_habits_overview`) does NOT expose `member_ids` — it returns `members_total: number` only. There's no client-side data source for the IN clause without an extra round-trip per habit.
- **Fix:** Filter narrowed to `user_id=eq.${userId}` (caller's own check-ins). The plan §<action> example also fell back to this same narrower filter ("Realtime can simply filter on user_id IN (<caller>)"). Co-member group-habit updates surface via the next manual refetch (screen focus / pull-to-refresh) — the additional latency is acceptable given the free-tier connection budget would otherwise require a per-group channel.
- **Files modified:** `src/hooks/useHabits.ts`
- **Verification:** useHabits.test.ts "subscribes to ONE per-user Realtime channel" assertion still passes; channel name `user-${userId}-checkins` includes the userId so distinct callers don't collide.
- **Committed in:** `4ba5a55`
