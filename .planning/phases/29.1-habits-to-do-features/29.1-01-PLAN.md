**D. Enable RLS + policies on all 6 tables** (clone shape from `0015:88-112` and `0017:146-168`):
- `habits`: SELECT for members via `is_habit_member`; INSERT by `created_by = auth.uid()`; UPDATE by creator only; DELETE by creator only.
- `habit_members`: SELECT for members (`is_habit_member`); INSERT done only via RPC (no client policy); UPDATE `accepted_at` only by `user_id = auth.uid()`; DELETE by self.
- `habit_checkins`: SELECT for members (`is_habit_member`); INSERT/DELETE only via `toggle_habit_today_checkin` RPC (no direct client write — restrict INSERT/DELETE to `user_id = auth.uid() AND is_habit_member(habit_id)`).
- `todos`: SELECT/INSERT/UPDATE/DELETE all gated by `user_id = auth.uid()` (personal — no helper needed).
- `chat_todo_lists`: SELECT for `created_by = auth.uid() OR assignee_id = auth.uid()`; INSERT only via RPC (no client INSERT policy).
- `chat_todo_items`: SELECT for `is_chat_todo_list_owner` OR list creator; UPDATE by assignee via RPC only.
- `messages`: **DO NOT add or remove existing policies**, but ensure the existing INSERT policy still allows the SECURITY DEFINER RPC to insert system rows — RPCs bypass RLS by virtue of being SECURITY DEFINER. Add an explicit client INSERT policy that blocks `message_type = 'system'` to prevent client spoofing (mitigates threat T-29.1-03 below):
```sql
DROP POLICY IF EXISTS messages_insert_policy ON public.messages;
CREATE POLICY messages_insert_policy ON public.messages FOR INSERT TO authenticated
  WITH CHECK (
    sender_id = (SELECT auth.uid())
    AND message_type IN ('text', 'image', 'poll')   -- 'system' is RPC-only
  );
```

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| client → Supabase RPC | Authenticated user supplies p_member_ids, p_assignee_id, p_items (jsonb) — all untrusted input |
| client → messages INSERT | Authenticated user could attempt to insert `message_type='system'` directly via PostgREST to spoof a "✓ X completed Y" notification |
| client → habit_checkins INSERT | Authenticated user could attempt to mark someone else done by varying `user_id` |
| RPC → RLS policy | SECURITY DEFINER bypasses RLS — every RPC must re-verify caller's access |
| habit_members → habit_members SELECT | Naïve RLS policy referencing same table triggers infinite recursion |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-29.1-01 | Spoofing | `assign_chat_todo` (D-10) — attacker assigns to-do to non-member user | mitigate | `create_chat_todo_list` RPC body re-verifies `p_assignee_id` is in `group_channel_members` for `p_group_channel_id`; RAISE EXCEPTION 'assignee not in chat' on mismatch. Verified by acceptance criteria string match. |
