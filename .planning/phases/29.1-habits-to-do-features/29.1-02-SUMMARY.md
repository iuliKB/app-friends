## Decisions Made

- **`MyTodoRow` as standalone interface (not `Todo & {...}` intersection).** The plan's draft showed `type MyTodoRow = Todo & { is_overdue, is_due_today }`, but the live `get_my_todos` RPC does NOT return `user_id` (the row is implicitly scoped to the caller). Intersecting with `Todo` would have falsely promised a `user_id: string` field at the type level that doesn't exist in the payload. Restating the open fields explicitly avoids the silent mismatch.
- **`ChatTodoRow` shape follows the live RPC, not the plan's draft.** Plan §`<action>` showed `{ list_id, message_id, group_channel_id, chat_name, title, is_list, done_count, total_count, earliest_due_date, created_at }`. The live `get_chat_todos` RPC (migration 0024 lines 808-866) instead returns `{ list_id, group_channel_id, message_id, created_by, title, is_list, created_at, total_count, done_count, next_due_date, is_overdue, is_due_today }` — no `chat_name`, `next_due_date` instead of `earliest_due_date`, plus the derived `is_overdue` / `is_due_today` booleans. Type matches the actual payload (Rule 1 — correctness).
- **`HabitOverviewRow` does NOT include `accepted_total`.** Live `get_habits_overview` returns `accepted_total: int` in addition to `members_total: int`, but the plan's explicit field list does not mention it. TypeScript silently tolerates the excess runtime field — kept the contract conservative and will add when the first hook consumer needs to distinguish accepted vs total. Documented inline in `habits.ts`.
- **`is_solo` semantics aligned to live RPC.** Plan comment said `is_solo` is "true when members_total === 1 AND only member is caller". Live RPC computes `(COALESCE(mc.accepted_total, 1) = 1) AS is_solo` — i.e., true when there's exactly one accepted member. Comment in `habits.ts` rewritten to match the live semantics ("true when accepted_total === 1 (only the caller)") to avoid hooks misinterpreting the flag.

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 1 — Bug] `ChatTodoRow` shape rewritten to match live `get_chat_todos` RPC payload**
- **Found during:** Task 1 (reading migration 0024 lines 808-866 before writing the type).
- **Issue:** Plan's `<action>` block specified `ChatTodoRow` with fields `chat_name: string` and `earliest_due_date: string | null`, but the live RPC does not return either of these. It returns `next_due_date`, `is_overdue`, `is_due_today`, and `created_by` instead. A TypeScript type that doesn't match the runtime payload silently leaves consumers with `undefined` reads, defeating the whole point of a contract layer.
- **Fix:** `ChatTodoRow` interface now has `{ list_id, group_channel_id, message_id, created_by, title, is_list, created_at, total_count, done_count, next_due_date, is_overdue, is_due_today }` — exact match to the live RPC return.
- **Files modified:** `src/types/todos.ts`
- **Verification:** Compared field-by-field against `supabase/migrations/0024_habits_todos_v1_8.sql:808-866`. `tsc --noEmit --strict` passes.
- **Committed in:** `ce381a2`

**2. [Rule 1 — Bug] `MyTodoRow` redeclared as standalone interface (not Todo intersection)**
- **Found during:** Task 1.
- **Issue:** Plan showed `type MyTodoRow = Todo & { is_overdue: boolean; is_due_today: boolean }`. The intersection would promise a `user_id: string` field (from `Todo`) that the live RPC does NOT return. Hooks would read `myTodoRow.user_id` and get `undefined` at runtime.
- **Fix:** `MyTodoRow` declared as a standalone interface listing only the fields the RPC actually returns (`id`, `title`, `due_date`, `notes`, `priority`, `completed_at`, `created_at`, `is_overdue`, `is_due_today`).
- **Files modified:** `src/types/todos.ts`
- **Verification:** Compared field-by-field against `supabase/migrations/0024_habits_todos_v1_8.sql:764-798`.
- **Committed in:** `ce381a2`
