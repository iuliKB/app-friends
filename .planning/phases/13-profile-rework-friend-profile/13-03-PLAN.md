<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| User device → Supabase effective_status view | Status query crosses this boundary; view enforces freshness |
| User device → Supabase profiles table | Birthday fields fetched via SELECT; RLS enforces friend visibility |
| User device → Supabase wish_list_items | useFriendWishList fetches friend's items; RLS enforces read-only access for non-owners |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-13-03-01 | Information Disclosure | profiles SELECT birthday_month/birthday_day | accept | RLS on profiles already allows friends to read birthday fields — this was verified in v1.4 (birthday feature shipped). Adding columns to SELECT does not change the access model. |
| T-13-03-02 | Spoofing | effective_status view returning stale status | mitigate | Using effective_status view (not statuses table) ensures freshness — view computes TTL expiry. NULL result treated as no active status, status row omitted. |
| T-13-03-03 | Information Disclosure | useFriendWishList claim visibility | accept | RLS on wish_list_claims hides claims from the item owner (birthday-secrecy feature). On friend profile, readOnly={true} means no claim UI is rendered regardless. The hook returns isClaimed/isClaimedByMe but these are not displayed in readOnly mode. |
| T-13-03-04 | Tampering | wishListItems.map WishListItem | accept | WishListItem with readOnly={true} renders no interactive controls. No mutation is possible from this surface. toggleClaim from the hook is not called in this context. |
</threat_model>
