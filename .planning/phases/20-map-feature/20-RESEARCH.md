# Phase 20: Map Feature - Research

**Researched:** 2026-04-29
**Domain:** React Native Maps, Expo Location, Supabase schema migration, deep link navigation
**Confidence:** HIGH

## Summary

This phase adds map-based location attachment to plans and a map view to the Explore tab. The technical foundation is two libraries — `react-native-maps` (MapView, Marker) and `expo-location` (foreground permission, getCurrentPositionAsync, reverseGeocodeAsync) — neither of which is currently installed. Both require plugin entries in `app.config.ts` and a new EAS build.

A critical SDK 55 compatibility issue exists: `react-native-maps` breaks when using **Google Maps on iOS** under Expo SDK 55 due to a broken AppDelegate config plugin. This project already decided (D-04, v1.6 pre accumulated context) to use Apple Maps (PROVIDER_DEFAULT) on iOS, which avoids the broken code path entirely. Android uses Google Maps, which requires a new API key to be provisioned and wired in via `EXPO_PUBLIC_GOOGLE_MAPS_KEY`.

The database change is small: two nullable `FLOAT8` columns (`latitude`, `longitude`) added to the `plans` table in migration `0020`. All existing plans (null lat/lng) are unaffected. The `location TEXT` column stays and becomes the reverse-geocoded address label populated exclusively via `expo-location reverseGeocodeAsync`.

**Primary recommendation:** Install `react-native-maps` + `expo-location`, add both plugins to `app.config.ts`, provision the Android Google Maps API key, run migration `0020`, then build the three UI surfaces (LocationPicker modal, PlanDashboard tile, ExploreMapView) in separate waves.

### Pattern 1: Map-Drag-Not-Pin-Drag (LocationPicker)

**What:** Fixed crosshair/pin overlaid on MapView; user drags the map underneath. `onRegionChangeComplete` captures new center coordinates.

**When to use:** Anytime the user needs to pick a precise map location. Avoids the UX friction of drag-a-small-pin.

**Example:**
```tsx
// Source: react-native-maps documentation + community pattern (VERIFIED: github.com/react-native-maps/react-native-maps/issues/1542)
const [region, setRegion] = useState<Region>(initialRegion);

<View style={{ flex: 1 }}>
  <MapView
    style={StyleSheet.absoluteFillObject}
    initialRegion={initialRegion}
    onRegionChangeComplete={(r, { isGesture }) => {
      if (isGesture) setRegion(r);
    }}
  />
  {/* Fixed pin overlay — absolutely positioned at center */}
  <View style={styles.pinContainer} pointerEvents="none">
    <Ionicons name="location" size={36} color={colors.accent.primary} />
  </View>
</View>
```

The `isGesture` guard prevents infinite update loops if the region is set programmatically.

### Pattern 2: expo-location Permission + GPS + Reverse Geocode

**What:** Request foreground permission, get current position, reverse geocode to address string.

**When to use:** On LocationPicker mount to center the map on user's GPS. On confirm to generate the address label.

**Example:**
```tsx
// Source: [CITED: docs.expo.dev/versions/latest/sdk/location/]
import * as Location from 'expo-location';

async function requestAndGetLocation() {
  const { status } = await Location.requestForegroundPermissionsAsync();
  if (status !== 'granted') return null;
  return Location.getCurrentPositionAsync({ accuracy: Location.Accuracy.Balanced });
}

async function reverseGeocode(lat: number, lng: number): Promise<string> {
  const results = await Location.reverseGeocodeAsync({ latitude: lat, longitude: lng });
  const r = results[0];
  // Recommended format: "123 Main St, Toronto" — streetNumber + street + city
  const parts = [
    r.streetNumber,
    r.street,
    r.city,
  ].filter(Boolean);
  return parts.join(', ') || r.formattedAddress || `${lat.toFixed(4)}, ${lng.toFixed(4)}`;
}
```

`LocationGeocodedAddress` fields returned: `city`, `country`, `district`, `formattedAddress`, `isoCountryCode`, `name`, `postalCode`, `region`, `street`, `streetNumber`, `subregion`, `timezone` (iOS only). [CITED: docs.expo.dev/versions/latest/sdk/location/]

### Pattern 4: Native Maps Deep Link (MAP-05)

**What:** Platform-conditional URL to open the device's preferred maps app.

```ts
// Source: [CITED: reactnative.dev/docs/linking]
import { Linking, Platform } from 'react-native';

export async function openInMapsApp(lat: number, lng: number, label: string): Promise<void> {
  const encodedLabel = encodeURIComponent(label);
  const url = Platform.OS === 'ios'
    ? `maps://?ll=${lat},${lng}&q=${encodedLabel}`
    : `geo:${lat},${lng}?q=${lat},${lng}(${encodedLabel})`;
  const canOpen = await Linking.canOpenURL(url);
  if (canOpen) {
    await Linking.openURL(url);
  }
}
```

For MAP-05's "Apple Maps / Google Maps / Waze" requirement, `react-native-map-link` `showLocation()` presents an action sheet letting the user pick. This is preferable over hard-coding the URL scheme if the goal is true user choice. [CITED: github.com/tschoffelen/react-native-map-link]

### Pattern 5: Haversine Distance (Explore 25km Filter)

```ts
// Source: [ASSUMED - standard formula, multiple sources confirm implementation]
export function haversineKm(
  lat1: number, lng1: number,
  lat2: number, lng2: number
): number {
  const R = 6371;
  const dLat = ((lat2 - lat1) * Math.PI) / 180;
  const dLng = ((lng2 - lng1) * Math.PI) / 180;
  const a =
    Math.sin(dLat / 2) ** 2 +
    Math.cos((lat1 * Math.PI) / 180) *
    Math.cos((lat2 * Math.PI) / 180) *
    Math.sin(dLng / 2) ** 2;
  return R * 2 * Math.atan2(Math.sqrt(a), Math.sqrt(1 - a));
}
```

### Anti-Patterns to Avoid

- **Using Google Maps on iOS (`PROVIDER_GOOGLE`):** Broken in SDK 55 — the `react-native-maps` config plugin cannot patch the AppDelegate correctly. Already decided against in D-04 and v1.6 pre context. [VERIFIED: github.com/expo/expo/issues/42423]
- **Putting pin as a Marker inside MapView for the LocationPicker:** The correct pattern is a fixed overlay View (pin graphic) positioned at screen center, not a draggable Marker. Markers are hard for users to grab precisely.
- **Setting `region` prop (controlled) on a MapView with `onRegionChangeComplete`:** Use `initialRegion` for the LocationPicker so the view is uncontrolled; only update state from gesture events. Mixing controlled `region` + gesture callbacks causes update storms.
- **Calling reverseGeocodeAsync before requesting permission:** On Android, calling geocoding APIs before the permission grant causes the app to hang indefinitely. Always check/request permission first. [CITED: github.com/expo/expo/issues/636]
- **Using `<MapView>` inside a ScrollView without `pointerEvents="none"`:** The MapView scroll gesture will intercept the parent ScrollView's scroll. Wrap the static tile with `pointerEvents="none"` or use a `<TouchableOpacity>` wrapper that absorbs taps.
- **Hardcoding COLORS in map components:** All new components must follow the Phase 19 pattern: `StyleSheet.create` inside `useMemo([colors])`, `useTheme()` for all color references. [VERIFIED: STATE.md accumulated context]

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Opening maps app on iOS + Android | Custom URL scheme handler | `Linking.openURL` with platform URL OR `react-native-map-link` | URL schemes differ per platform and per app (Apple Maps, Google Maps, Waze); `react-native-map-link` handles all variants and fallbacks |
| Geocoding address from coordinates | HTTP call to Google Geocoding API | `expo-location reverseGeocodeAsync` | OS-native, free, no API key, works offline — already decided D-11 |
| Distance calculation | PostGIS query or external API | Inline haversine function | 25km filter on a small plan list (< 100 items) is trivial in JS |

**Key insight:** The OS-native geocoder is good enough for address labels at this scale; paid geocoding APIs add cost and complexity for no benefit.

## Common Pitfalls

### Pitfall 1: SDK 55 + Google Maps iOS AppDelegate conflict
**What goes wrong:** Installing `react-native-maps` with `iosGoogleMapsApiKey` set in the plugin causes "Cannot add Google Maps to the project's AppDelegate because it's malformed" at EAS build time.
**Why it happens:** Expo SDK 55 changed the AppDelegate structure; the `react-native-maps` config plugin for iOS Google Maps hasn't been updated.
**How to avoid:** Do NOT set `iosGoogleMapsApiKey` in the plugin config. iOS uses Apple Maps (PROVIDER_DEFAULT). [VERIFIED: github.com/expo/expo/issues/42423, github.com/react-native-maps/react-native-maps/issues/5843]
**Warning signs:** EAS build error referencing AppDelegate; never appears in Expo Go.

### Pitfall 2: reverseGeocodeAsync hangs on Android if called before permission
**What goes wrong:** The app freezes indefinitely when `reverseGeocodeAsync` is called without a prior permission grant.
**Why it happens:** Android geocoder requires location permission even for reverse geocoding. [CITED: github.com/expo/expo/issues/636]
**How to avoid:** Always call `requestForegroundPermissionsAsync` and confirm status === 'granted' before any location or geocoding call.
**Warning signs:** LocationPicker appears to freeze after confirming a location.

### Pitfall 3: MapView inside ScrollView intercepts scrolls
**What goes wrong:** On PlanDashboard, the non-interactive map tile prevents the user from scrolling the screen — touch events go to the MapView.
**Why it happens:** MapView's native gesture handler captures touch events even when scroll/zoom are disabled.
**How to avoid:** Wrap the map tile in a `<View pointerEvents="none">` so all touch events pass through to the parent ScrollView.
**Warning signs:** The scroll sticks when the user swipes over the map tile area.

### Pitfall 4: `onRegionChangeComplete` infinite loop
**What goes wrong:** Setting region state in `onRegionChangeComplete` triggers a re-render which triggers another region change, looping.
**Why it happens:** Controlled `region` prop causes the MapView to fire `onRegionChangeComplete` again when updated programmatically.
**How to avoid:** Use `initialRegion` (uncontrolled) instead of `region` for the LocationPicker. Only update the local coordinate state, not the map's region directly. Guard with `isGesture` flag. [VERIFIED: react-native-maps GitHub issue #1542]
**Warning signs:** Excessive re-renders visible in React DevTools; map jitters.

### Pitfall 5: EAS rebuild required after adding native plugins
**What goes wrong:** Running the app in Expo Go or a pre-existing development build after adding `react-native-maps` results in "Invariant Violation: Native module MapView not found".
**Why it happens:** `react-native-maps` contains native code; Expo Go does not include it; a new EAS development build is required.
**How to avoid:** The plan must include an EAS development build step immediately after installing packages and updating `app.config.ts`. No amount of JS restarts will fix a missing native module.
**Warning signs:** Runtime crash referencing MapView or null native module.

### Pitfall 6: `latitude`/`longitude` not included in Supabase select queries
**What goes wrong:** The Explore map view or PlanDashboard shows no map tile even after migration because `latitude` and `longitude` are null in the fetched data.
**Why it happens:** `usePlans` and `usePlanDetail` use `select('*')` which should include new columns — BUT the `usePlans` manual type-cast assembly (lines 128-140) explicitly maps only known columns. New columns must be added to the mapping.
**How to avoid:** After migration, add `latitude` and `longitude` to the `PlanWithMembers` assembly in `usePlans.ts` and to the `plan` mapping in `usePlanDetail.ts`.
**Warning signs:** `plan.latitude` is undefined (not null) after migration.

### Location Permission + GPS Center

```tsx
// Source: [CITED: docs.expo.dev/versions/latest/sdk/location/]
async function getInitialRegion(): Promise<Region> {
  const DEFAULT_REGION: Region = {
    latitude: 43.6532,   // Toronto — reasonable default for a Canadian-English app
    longitude: -79.3832,
    latitudeDelta: 0.05,
    longitudeDelta: 0.05,

  const { status } = await Location.requestForegroundPermissionsAsync();

  const pos = await Location.getCurrentPositionAsync({ accuracy: Location.Accuracy.Balanced });
  return {
    latitude: pos.coords.latitude,
    longitude: pos.coords.longitude,
    latitudeDelta: 0.01,
    longitudeDelta: 0.01,
  };
}
```

### Address Label Formatting (D-11 discretion resolved)

Recommended format: `"123 Main St, Toronto"` — combines street number, street name, and city. Falls back to `formattedAddress` if either part is null, and to raw coordinates as last resort.

```ts
function formatAddress(r: Location.LocationGeocodedAddress, lat: number, lng: number): string {
  const streetParts = [r.streetNumber, r.street].filter(Boolean).join(' ');
  const parts = [streetParts, r.city].filter(Boolean);
  if (parts.length > 0) return parts.join(', ');
  return r.formattedAddress ?? `${lat.toFixed(5)}, ${lng.toFixed(5)}`;
}
```

## Open Questions

1. **Android Google Maps API key provisioning**
   - What we know: A key must be created in Google Cloud Console restricted to Android Maps SDK and stored as `EXPO_PUBLIC_GOOGLE_MAPS_KEY` in the EAS secret store.
   - What's unclear: Whether the user already has a Google Cloud project set up (likely yes, given Supabase is configured).
   - Recommendation: Plan Wave 0 should include a clear step to provision this key; flag as a manual prerequisite step.

2. **react-native-map-link vs bare Linking.openURL for MAP-05**
   - What we know: `Linking.openURL` with `maps://` (iOS) / `geo:` (Android) covers the basic case. `react-native-map-link` adds an action sheet for app choice.
   - What's unclear: Whether the user wants a picker or prefers the OS default maps app.
   - Recommendation: Use `react-native-map-link` for better UX and to satisfy "Google Maps, Waze, or Apple Maps" language in MAP-05; it handles all platforms and edge cases.

3. **Location permission timing for ExploreMapView**
   - What we know: Foreground permission must be granted before reading GPS for the 25km filter.
   - What's unclear: Whether permission is already granted from the LocationPicker, or whether ExploreMapView opens first.
   - Recommendation: Always call `requestForegroundPermissionsAsync` idempotently at the start of both LocationPicker and ExploreMapView; the OS will only prompt once per session.

### Known Threat Patterns for This Stack

| Pattern | STRIDE | Standard Mitigation |
|---------|--------|---------------------|
| Spoofed GPS coordinates | Tampering | Coordinates come from on-device GPS/user pin; no server trust issue; lat/lng are informational, not access-control |
| Insecure deep link handling | Elevation of Privilege | `Linking.canOpenURL` guards before `openURL`; no user-supplied URL; destination is always constructed from DB-stored coordinates |
| Over-broad location permission | Info Disclosure | Foreground-only (`requestForegroundPermissionsAsync`); no background permission requested — enforced at config level |
| lat/lng leakage to unauthorized users | Info Disclosure | Existing Supabase RLS on `plans` table restricts reads to plan members; lat/lng columns inherit same policy |
