<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Client → Supabase RLS | Profile update goes through Supabase RLS UPDATE policy WITH CHECK (id = auth.uid()); only the authenticated user can update their own birthday |
| Client input → DB | birthday_month and birthday_day values come from a controlled dropdown list; no free-text input, so injection is not possible |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-06-04 | Tampering | handleSave — birthday_month value | mitigate | Values sourced only from BirthdayPicker dropdown (1-12); no free-text path; DB CHECK constraint is the final server-side gate |
| T-06-05 | Tampering | handleSave — birthday_day value | mitigate | Values sourced only from BirthdayPicker dropdown (bounded by getDaysInMonth); partial birthday guard clears both if one is null; DB compound CHECK is server-side gate |
| T-06-06 | Tampering | handleSave — write another user's birthday | accept | RLS UPDATE policy WITH CHECK (id = auth.uid()) already in place from migration 0016 — no client code needed |
| T-06-07 | Information Disclosure | birthday_year exposure | accept | No year is ever collected, stored, or displayed — only month+day; birthday year is never in the data model |
| T-06-08 | Elevation of Privilege | Unauthenticated save | accept | edit.tsx early-returns if session is null before any Supabase call; useAuthStore enforces session presence |
</threat_model>
