<objective>
Create the value-led iOS pre-prompt modal component (D-02 copy) and wire the eligibility flag to fire ONCE the user completes their first meaningful action — defined per D-01 as either (a) setting their own status, or (b) adding their first friend. This satisfies PUSH-08.

Purpose: iOS only allows the system permission modal to be shown ONCE per install lifetime. Asking eagerly on first launch wastes that one-shot. The pre-prompt UI is a soft ask that the user can decline without burning the OS-level prompt; the eligibility flag ensures we don't even ask the soft question until the user has given a signal that they care about Campfire.

Output: New `PrePromptModal.tsx` component, plus targeted edits to `useStatus.ts` and `useFriends.ts` adding a single `markPushPromptEligible()` call after their first successful mutation.

Note on rendering the modal: The PrePromptModal component is created in this plan but the call site that *displays* it (controlled by a parent screen with visible state) is intentionally minimal in v1.3 — the modal can be rendered from `(tabs)/_layout.tsx` triggered when `registerForPushNotifications` returns `'not_eligible'` AND eligibility just flipped to true. For Phase 1 scope, the modal component + the eligibility flag wiring are sufficient to unblock Plan 04's session-ready effect, which will naturally surface the modal on the next AppState 'active' transition after the user sets a status. Detailed render-trigger UX can be tuned during smoke testing (Plan 10) without re-touching the modal itself.
</objective>
