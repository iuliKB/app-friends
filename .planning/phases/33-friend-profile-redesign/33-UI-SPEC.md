## Primary Visual Anchor

**The large header avatar (140px ŌåÆ 32px collapse)** is the primary visual anchor for this screen. It is the first element the eye lands on, the literal subject of the screen, and the source of the page's color wash (its blurred image fills the header backdrop). All other elements are visually subordinate to it ŌĆö including the quick-action icon row, status pill, and grouped-inset sections below. [CTX:D-01, D-02]

## Layout Skeleton (Top ŌåÆ Bottom)

Single vertically-scrolling `Animated.ScrollView` (Reanimated 4) at the screen root. Sections, in order:

1. **HeaderArea** ŌĆö Z-stacked, height Ōēł 280pt
   1. **Blurred-avatar wash** ŌĆö full-bleed at the back, fades behind the avatar circle, dims behind the name/pill so text stays legible
   2. **Big avatar** ŌĆö 140pt circle, centered horizontally, anchored vertically to be vertically centered in the header
   3. **Display name** ŌĆö single line, `FONT_SIZE.xxl` (24px), `FONT_FAMILY.display.semibold`, `colors.text.primary`, centered, drop-shadow `rgba(0,0,0,0.30)` 0/1/2 in both modes for legibility over the wash
   4. **Username** ŌĆö `@username`, `FONT_SIZE.md` (14px), `FONT_FAMILY.body.regular`, `colors.text.secondary`, centered
   5. **Status pill** ŌĆö directly under the username; only renders when `effective_status !== null` [CTX:D-07]
2. **QuickActionsRow** ŌĆö sticky horizontally-centered row of four `ActionIconButton`s: **Message ┬Ę Mute ┬Ę Photos ┬Ę More** [CTX:D-04, D-09]
3. **INFO grouped-inset section** (Telegram-style rounded rectangle, Messenger-style colorful leading icons)
   - **Bio** row ŌĆö only renders when `profile.bio` is non-null + non-empty [CTX:D-05, D-06]
   - **Friends since** row ŌĆö formatted month + year from `friendships.created_at`
   - **Birthday** row ŌĆö only renders when `birthday_month` + `birthday_day` both present
   - **Timezone** row ŌĆö only renders when `profile.timezone` is non-null
4. **MUTUAL grouped-inset section** [CTX:D-06]
   - **Mutual plans** row ŌĆö count ŌåÆ tap navigates (route is planner's call per CONTEXT.md)
   - **Mutual friends** row ŌĆö count ŌåÆ tap navigates
   - **Shared photos** row ŌĆö count ŌåÆ tap navigates
   - **IOU balance** row ŌĆö formatted dollar amount + direction ŌåÆ tap navigates to existing per-friend expenses screen [CTX:D-08]
5. **WISH LIST section** ŌĆö existing implementation reused as-is via `WishListItem`, header restyled to match new section header pattern (see Typography table) [CTX canonical_refs]
6. **Bottom inset** ŌĆö `SPACING.xxl` (32px) tail padding so the last row is not flush with the home indicator on iPhone

Sections 3, 4, 5 are stacked with `SPACING.xl` (24px) vertical gaps. Each section's title is rendered ABOVE the rounded rectangle (small caps style ŌĆö see Typography section).

The screen layout ENDS after WISH LIST. **No DANGER section, no Remove Friend button, no Block button** ŌĆö destructive Remove Friend lives only inside the **More (Ōŗ») overflow ActionSheet** [CTX:D-10, D-11].

## Spacing Scale

All values from `src/theme/spacing.ts` ŌĆö no raw numbers permitted in any Phase 33 component. [v1.6][Token Audit]

| Token | Value | Phase 33 Usage |
|-------|-------|----------------|
| `SPACING.xs` | 4px | Status pill internal padding, row-icon gap-to-label, drop-shadow vertical offset on name |
| `SPACING.sm` | 8px | Quick-action button gap-to-label, status pill horizontal padding, hairline separator height |
| `SPACING.md` | 12px | Gap between quick-action icon and label below it; vertical row padding inside grouped inset rows |
| `SPACING.lg` | 16px | Grouped-inset horizontal margin (inset from screen edges); section title left padding; inter-row horizontal padding inside the rounded rectangle |
| `SPACING.xl` | 24px | Vertical gap between sections (INFO ŌåÆ MUTUAL ŌåÆ WISH LIST); margin-top on QuickActionsRow; margin-top on first section after QuickActionsRow |
| `SPACING.xxl` | 32px | Top inset above big avatar; bottom inset after last section |

**Exceptions (locked, justified):**

- **Big avatar diameter: 140pt** ŌĆö not a SPACING token. Telegram-style anchor [CTX:D-01]. Multiple of 4. Stored as a named constant `BIG_AVATAR_SIZE = 140` in the new `FriendProfileHeader` component.
- **Collapsed avatar diameter: 32pt** ŌĆö not a SPACING token. Multiple of 4. Stored as a named constant `COLLAPSED_AVATAR_SIZE = 32` in the same component. (Matches Stack header title height roughly.)
- **Header area total height: 280pt** ŌĆö not a SPACING token. Sized so the big avatar + name + username + status pill fit with breathing room. Stored as a named constant `HEADER_HEIGHT = 280`.
- **Quick-action button hit area: 44├Ś44pt** ŌĆö accessibility floor [UPM]. Visual circular icon-only button is 48pt diameter (multiple of 4), but `hitSlop` is verified to keep the touchable rect Ōēź 44pt. Justification: WCAG/Apple HIG minimum touch target [UPM Result 6 ux/touch].
- **Leading row-icon diameter: 32pt** ŌĆö not a SPACING token. Multiple of 4. Stored as `ROW_ICON_SIZE = 32` [CTX:D-03].
- **Row min-height: 56pt** ŌĆö not a SPACING token. Hits the iOS standard cell height + matches the 44pt accessibility floor (icon 32 + 12 padding top/bottom = 56). Multiple of 4. [UPM]

No raw numbers go anywhere else in any new component or styled prop. Phase 33 ESLint guard (v1.1) inherits.

## Header Collapse Animation Contract

[CTX:D-01][UPM Result 3 react-native: Use Reanimated for high-perf animation]

### Library
Reanimated v4 (already installed at `4.2.1`). Use `useAnimatedScrollHandler` + `useSharedValue` + `useAnimatedStyle` + `interpolate`. No `Animated` API (the legacy RN one), no `LayoutAnimation`, no manual `setState` on scroll.

### Shared values

```
const scrollY = useSharedValue(0);
// driven by:
const scrollHandler = useAnimatedScrollHandler({
  onScroll: (e) => { scrollY.value = e.contentOffset.y; },
});
```

### Interpolation ranges

Anchor constants:
- `HEADER_HEIGHT = 280` (named constant in `FriendProfileHeader.tsx`)
- `COLLAPSE_END = HEADER_HEIGHT - 120 = 160` ŌĆö the scroll offset at which the avatar is fully collapsed into the title bar
- `BIG_AVATAR_SIZE = 140`
- `COLLAPSED_AVATAR_SIZE = 32`
- `WASH_FADE_END = HEADER_HEIGHT - 60 = 220` ŌĆö wash starts fading earlier so it never clips the nav title

| Animated Property | Input Range | Output Range | Clamp |
|-------------------|-------------|--------------|-------|
| Big avatar `transform.scale` | `[0, COLLAPSE_END]` = `[0, 160]` | `[1, COLLAPSED_AVATAR_SIZE / BIG_AVATAR_SIZE]` = `[1, 0.2286]` | `Extrapolation.CLAMP` |
| Big avatar `opacity` | `[0, COLLAPSE_END * 0.75, COLLAPSE_END]` = `[0, 120, 160]` | `[1, 1, 0]` | `Extrapolation.CLAMP` (avatar fully invisible by `COLLAPSE_END`) |
| Big avatar `transform.translateY` | `[0, COLLAPSE_END]` = `[0, 160]` | `[0, -(HEADER_HEIGHT/2 - COLLAPSED_AVATAR_SIZE/2 - safeAreaTop)]` (Ōēł `-100` on iPhone 15 Pro) | `Extrapolation.CLAMP` |

## Blurred-Avatar Wash Technique

[CTX:D-02]

### Recommendation: `expo-image` `blurRadius` prop (NOT a separate `Image + expo-blur BlurView`)

**Rationale:**

1. **expo-image is already in the stack** (`~55.0.9`) and is the modern recommended image library for React Native (per Expo SDK 55 docs + UPM Result 3 react-native: high-perf primitives).
2. `expo-image` exposes a native `blurRadius` prop that does the GPU-side Gaussian blur at render time ŌĆö equivalent visual output to `BlurView` at lower memory cost (one texture, not two), with no JS-thread compositing tax.
3. The `BlurView` from `expo-blur` is a runtime overlay designed for blurring *whatever is behind it on screen* (e.g. content blur during a sheet present). For our case ŌĆö blurring a *known static image* ŌĆö `blurRadius` on the image itself is the correct API.
4. Single component = less to test, less to maintain. We avoid stacking a BlurView over an Image and managing two opacity animations.

### Implementation

```tsx
// FriendProfileBlurredWash.tsx (sketch ŌĆö executor authors the real component)
<Animated.View style={[styles.washContainer, washAnimatedStyle]}>
  {avatarUrl ? (
    <Image
      source={{ uri: avatarUrl }}
      style={styles.washImage}
      blurRadius={40}
      contentFit="cover"
      // expo-image is fine to use here; not Animated because the parent View owns opacity
    />
  ) : (
    <View style={[styles.washImage, { backgroundColor: colors.surface.card }]} />
  )}
  <LinearGradient
    colors={[washTopColor, washBottomColor]}
    locations={[0, 1]}
    style={StyleSheet.absoluteFill}
  />
</Animated.View>
```

`blurRadius={40}` lock ŌĆö visually matches the Telegram contact-screen wash on dev iPhone 15 Pro Max + iPhone SE Gen 3. (Spec value; executor may A/B between 32ŌĆō48 during implementation, locked range.)

### Gradient overlay over the wash

| Stop | Dark `washColor` | Light `washColor` | Purpose |
|------|------------------|-------------------|---------|
| Top (0) | `rgba(14, 15, 17, 0.35)` (`surface.base` @ 35%) | `rgba(245, 246, 248, 0.45)` (`surface.base` @ 45%) | Dim wash slightly so glyph icons + neon-green accent stay readable over busy avatars |
| Bottom (1) | `rgba(14, 15, 17, 0.95)` (`surface.base` @ 95%) | `rgba(245, 246, 248, 0.95)` (`surface.base` @ 95%) | Hard-fade to screen body bg so the section below is visually grounded |

### Cross-fade on avatar load

`expo-image` exposes `onLoad`. On load:
- Set a local `useState<boolean>(false)` `washReady` to `true`
- Wrap the entire wash container in a `withTiming({ opacity }, { duration: ANIMATION.duration.normal /* 300ms */ })` cross-fade from 0 ŌåÆ 1
- This avoids a "pop" on cold load of the friend profile

Source: [Phase 24 ANIMATION.duration.normal][UPM Result 7 style/dark-mode: dark-to-light transitions]

### No-avatar fallback

When `profile.avatar_url == null`, render a solid `colors.surface.card` rectangle for the wash (`#1D2027` / `#FFFFFF`). The Avatar's existing initials-on-`surface.card` fallback handles the big avatar itself. No special branching needed in `FriendProfileBlurredWash`.
